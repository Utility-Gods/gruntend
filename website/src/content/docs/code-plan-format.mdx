---
title: Why code plans
description: Why Gruntend uses constrained JavaScript instead of a workflow DSL.
---

A Gruntend plan is JavaScript.

```js
const menus = await tools.menus.list({});
const items = await Promise.all(
  menus.menus.map((menu) => tools.menu.items.list({ menuId: menu.menuId })),
);

return items.flatMap((result) => result.items);
```

It runs as an async function body with a deliberately small environment:

```text
input
registered tools
Promise
console
html, when generated UI is enabled
```

It does not receive arbitrary imports, `fetch`, `window`, `document`, storage, or application internals.

## Why code?

Earlier Gruntend designs used declarative workflow JSON. That works for simple linear steps, but becomes awkward when behavior depends on runtime results:

- filter records using a value discovered at runtime
- branch based on a previous tool result
- fan out independent reads with `Promise.all()`
- loop over a selected set
- return early when nothing matches

Those are ordinary programming constructs—not edge cases. A code plan expresses them directly while the tool registry preserves the authority boundary.

```js
const menus = await tools.menus.list({});
const matching = menus.menus.filter((menu) =>
  menu.name.includes(input.query),
);

if (matching.length === 0) {
  return html`<p>No matching menus.</p>`;
}

return html`<ul>${matching.map(
  (menu) => html`<li>${menu.name}</li>`,
)}</ul>`;
```

## Input is supplied by the app

Runtime input is passed separately from the plan and exposed through the `input` global.

```ts
const result = await gruntend.runCodePlan(code, {
  input: {
    query: "Dinner",
  },
  handlers,
});
```

## Generate plans or bring your own

`generateCodePlan()` can ask a supported model to produce the JavaScript plan. You can also use your own model or agent loop and pass the resulting code into the same runtime.

Generation is optional. The constrained runtime, typed capabilities, app-owned handlers, and generated UI primitives are usable independently.
