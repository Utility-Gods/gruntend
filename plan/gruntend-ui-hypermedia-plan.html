Gruntend planning note · UI protocol research · 2026-07-03

Agentic Hypermedia over Application Capabilities

The frontend primitive should not be a chat window, Solid app, React component tree, or raw JSON UI DSL. The primitive should be a hypermedia representation of application state plus allowed next actions, backed by runtime-owned capabilities and opaque continuation handles.

semantic runtime hypermedia controls runtime authority boundary design-system friendly extension intercepts

Core thesis

Other projects are mostly trying to solve AI-generated UI. Gruntend should solve a slightly different problem: 

Applications expose semantic capabilities
↓
Runtime/agent produces hypermedia projections
↓
User follows safe controls
↓
Runtime validates and executes through app-owned tools

So the frontend does not receive raw tool authority. It receives a representation of state and allowed transitions. The UI may trigger capabilities, but only by submitting runtime-minted controls.

What people are trying in the ecosystem

1. Controlled generative UI

Examples: Vercel AI SDK, CopilotKit frontend tools, assistant-ui tool renderers.

model calls tool
↓
tool returns data
↓
developer maps result to known component

Strength: safest and easiest to reason about.

Weakness: app must prebuild every renderer; mostly chat/tool-result oriented.

2. Declarative component protocols

Examples: A2UI, json-render, LangChain generative UI with json-render.

developer defines catalog
↓
LLM generates JSON UI tree
↓
renderer maps to design system

Strength: framework-agnostic, streamable, design-system friendly.

Weakness: primarily component-centered, not application-capability-centered.

3. Sandboxed HTML / app iframe

Examples: MCP Apps, mcp-ui, OpenAI Apps SDK, OpenGenerativeUI.

tool links/returns HTML app
↓
host renders iframe
↓
iframe talks through bridge

Strength: extremely flexible for charts, editors, dashboards, maps, simulations.

Weakness: heavy security boundary: iframe sandbox, CSP, postMessage bridge, permissions.

4. Agent/user event protocols

Example: AG-UI.

run started
text chunks
tool call events
state deltas
interrupts
run finished

Strength: strong session/event/lifecycle layer.

Weakness: not itself the semantic hypermedia projection format.

5. Markdown/custom block renderers

Example: llm-ui.

LLM streams markdown
↓
parser detects custom blocks
↓
React renders matched blocks

Strength: good chat polish and progressive content handling.

Weakness: too presentation-oriented for Gruntend's semantic runtime.

Recommended Gruntend primitives

Session

The active runtime context. Owns lifecycle, event stream, action registry, pending requests, and projection state.

Surface

A renderable hypermedia region: state representation plus allowed next controls. A chat message, drawer, modal, card, table, or command palette can all render the same surface.

Projection

The semantic view of app/runtime state: entity lists, forms, approvals, trace, results, errors, and choices.

Control

A user-facing transition such as a button, choice, form submit, approval, or item action. It should be opaque to the renderer.

ActionRegistry

Session-local mapping from opaque action IDs to trusted operations or continuations. It lives wherever the runtime authority lives.

Renderer

Presentation layer only. It maps surfaces/controls to the host app's design system and calls submitAction(actionId).

How action mapping works without assuming a server

Do not say “server stores a_123.” Say “the runtime authority boundary stores a_123.” That boundary may be browser-local, server-backed, edge-backed, desktop-process-backed, worker-backed, or hybrid.

1. Draft affordance: runtime, model, or extension proposes “delete this menu item”.
2. Materialization: runtime validates the proposal against tool registry, entity state, permissions, and policy.
3. Mint handle: runtime creates a_123 in the current session action registry.
4. Emit surface: renderer receives only an opaque control, not the underlying tool name or privileged input.
5. Submit: user clicks; renderer calls session.submitAction("a_123").
6. Resolve: runtime revalidates and executes the mapped operation or resumes the plan.
ActionRegistry
  a_123 → {
    kind: "toolCall",
    tool: "menu.delete",
    input: { itemId: "item_1" },
    risk: "destructive",
    requiresApproval: true
  }
Rule: the UI should normally call submitAction(actionId), not tools.menu.delete(...). If tools are intentionally exposed in the browser, they are public client capabilities, not protected authority.

Hypermedia shape

Hypermedia means a representation includes the allowed next transitions. In Gruntend terms: 

Surface = representation of semantic application state
        + controls the user is allowed to follow
        + opaque continuation handles minted by runtime

The canonical protocol can later be serialized as semantic HTML, JSON AST, A2UI, json-render, or an MCP Apps iframe. The important part is that the authority mapping remains in the runtime session.

Possible wire/render formats

Semantic HTML fragment

Web-native and LLM-friendly. Can use native forms, buttons, lists, and tables. Needs sanitization and canonicalization.

<button gr-href="/menus/menu_1/items/item_2/actions/delete" data-gr-risk="destructive">
  Delete
</button>

JSON/AST surface

Easier to validate, transform, and map to non-web renderers. Less web-native than HTML.

{
  "kind": "entityList",
  "entityType": "menuItem",
  "controls": [{ "actionId": "a_123", "label": "Delete" }]
}

Adapter formats

Could emit A2UI/json-render/MCP Apps later from the same semantic surface model.

Gruntend Surface
↓
A2UI adapter | json-render adapter | HTML adapter | iframe adapter

Extension intercept model

Extensions should be able to participate, but only at explicit boundaries.

tool result
↓ projection intercept
validated/canonical surface
↓ render intercept
user action
↓ action/policy intercept
tool call / plan continuation
↓ tool-call intercept
new projection
Intercept Purpose Authority
Projection intercept Convert tool results into surfaces, remove unsupported controls, add metadata Must be revalidated
Render intercept Map semantic controls to host design system Presentation only
Action intercept Allow/deny, require approval, apply UI policy Runtime-side
Tool-call intercept Audit, permission checks, destructive-action confirmation Runtime-side authority

v0 recommendation

  1. Do not start with a drop-in Solid app as the core API.
  2. Define a framework-agnostic Session + Surface + Control contract.
  3. Keep tool handlers owned by the app/runtime, not the renderer.
  4. Materialize all user-facing actions into opaque session action IDs.
  5. Provide a simple renderer adapter for the example app.
  6. Add richer HTML/iframe extension support later, behind explicit sandbox/bridge boundaries.
Positioning: Gruntend is not “LLM generates frontend.” It is “applications expose capabilities, runtime projects hypermedia, user follows safe controls.”

Things to avoid early

  • Giving frontend renderers raw access to protected tool handlers.
  • Treating server as mandatory in the protocol vocabulary.
  • Making Solid/React/Vue the core abstraction.
  • Letting LLM-generated HTML mint executable authority directly.
  • Inventing a large layout DSL before the semantic action model is stable.
  • Mixing component generation concerns with the core tool contract.

Research notes

  • A2UI: JSON streaming UI protocol with surfaces, component catalog, data model, binding, actions, and renderer capability exchange.
  • json-render: catalog + registry model; AI generates flat JSON specs constrained to developer-defined components/actions; supports React/Vue/Svelte/Solid and more.
  • AG-UI: event protocol for agent/frontend interaction; useful lifecycle/event layer.
  • MCP Apps / mcp-ui / OpenAI Apps SDK: iframe app model where tools link to UI resources and widgets communicate through a host bridge.
  • Vercel AI SDK / assistant-ui / CopilotKit controlled UI: tool-call results rendered by known components; safe but less dynamic.
  • OpenGenerativeUI: high-flexibility generated HTML/SVG/JS inside sandboxed iframe; good extension pattern but not ideal as Gruntend core.
