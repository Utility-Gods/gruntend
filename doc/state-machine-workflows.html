GenOpen SDK design note

XState Workflow Runtime Direction

This document captures the current direction for consuming LLM-generated workflows in GenOpen. The application SDK separates tool contracts from runtime execution. The LLM should return an XState-compatible JSON machine, and GenOpen should bind that machine to registered tool contracts plus runtime handlers supplied by the application.

Current Decision

GenOpen should use XState as the workflow execution engine, not grow a custom workflow interpreter. The LLM response should be declarative machine JSON. GenOpen will validate the allowed subset, create an XState machine from it, provide a small set of actors, validate tool input/output with Standard Schema, and execute app-owned runtime handlers.

defineTool() is contract-only: name, description, input schema, output schema, and execution metadata. Runtime work is supplied later through closure handlers when the app triggers runWorkflow(). XState only affects the consumer side: how GenOpen runs the JSON workflow returned by the LLM.

Current Implemented Foundation

GenOpen currently has these foundational pieces:

Tool
↓
ToolRegistry
↓
runtime handlers
↓
runWorkflow

The current runWorkflow consumes an XState-style JSON object, invokes handlers for registered tools, follows onDone/onError, collects outputs, resolves $ref values, validates Standard Schema input/output contracts, and emits tool lifecycle events for UI previews.

What Does Not Change

The tool SDK does not need to change for XState. A tool is still the smallest domain primitive. The app defines tools; the LLM receives their contracts; the runtime executes those tools only through the registry. 

const createMenu = defineTool({
  name: "menu.create",
  description: "Create a menu",
  input: v.object({
    name: v.string()
  }),
  output: v.object({
    menuId: v.string(),
    name: v.string()
  }),
  execution: "sequential"
});

Runtime execution is supplied later, where request state, cookies, session data, component state, or database handles exist: 

await genopen.runWorkflow(workflowFromLLM, {
  handlers: {
    "menu.create": async ({ input }) => ({
      data: await api.createMenu(input, {
        token: authToken,
        tenantId
      })
    })
  }
});

The registry also stays simple:

const registry = createToolRegistry([
  createMenu,
  createMenuItem
]);

XState is not part of defining a tool. XState is part of consuming the workflow produced by the LLM.

Why XState

A workflow is not only a list of steps. It has states, transitions, success paths, failure paths, retries, parallel branches, final states, and eventually snapshots for pause/resume. These are state-machine concepts. If GenOpen keeps adding these concepts to a custom JSON format, it will slowly recreate a state machine engine.

Workflow concern XState concept
Run one tool after another State transition after an invoked actor completes
Run several tools concurrently Parallel states
Handle successful tool completion onDone
Handle tool failure onError
Retry or fallback Explicit retry/fallback states or safe retry metadata compiled by GenOpen
Finish workflow type: "final"
Pause/resume later Persisted XState snapshot

The Important Separation

The system has three separate contracts:

1. Tool Definition Contract
   App defines tools with input/output schemas.

2. Runtime Handler Contract
   App supplies closure handlers at workflow execution time.

3. Tool Manifest Contract
   GenOpen sends model-facing tool descriptions to the LLM.

4. Workflow Machine Contract
   LLM returns XState-compatible JSON that invokes registered tools.

The XState work only belongs to the third contract. It should not leak into the first contract.

What XState Does and What GenOpen Still Does

XState is well tested, but GenOpen still needs its own tested adapter layer. We are not testing XState internals. We are testing the bridge between LLM JSON, XState, and GenOpen tools.

XState owns GenOpen owns
State transitions Tool lookup in ToolRegistry
Invoke lifecycle Binding an XState actor to runtime handlers
onDone and onError routing Resolving $ref values from previous tool outputs
Parallel state semantics Collecting named tool outputs
Final state detection Validating the LLM emitted only the safe subset
Snapshots Persisting and restoring snapshots when that domain is added

Recommended LLM Workflow Shape

The first prototype used src: "tool:menu.create". For XState integration, a cleaner shape is to use one generic actor source, src: "tool", and put the concrete tool name in actor input.

Recommended invoke shape:

{
  "invoke": {
    "src": "tool",
    "input": {
      "tool": "menu.create",
      "params": {
        "name": "Dinner Menu"
      }
    },
    "onDone": {
      "target": "completed"
    },
    "onError": {
      "target": "failed"
    }
  }
}

This keeps XState actor registration simple. GenOpen registers one actor named tool. The actor reads input.tool, resolves input.params, looks up the tool in the registry, and executes it.

Example LLM-Generated Machine

This is the kind of JSON the LLM could return. It creates a menu, then creates one menu item using the menu id returned by the first tool. 

{
  "id": "menuWorkflow",
  "initial": "createMenu",
  "states": {
    "createMenu": {
      "invoke": {
        "src": "tool",
        "input": {
          "tool": "menu.create",
          "params": {
            "name": "Dinner"
          }
        },
        "onDone": {
          "target": "createBurger"
        },
        "onError": {
          "target": "failed"
        }
      }
    },
    "createBurger": {
      "invoke": {
        "src": "tool",
        "input": {
          "tool": "menu.item.create",
          "params": {
            "menuId": { "$ref": "createMenu.data.menuId" },
            "name": "Burger",
            "price": 12
          }
        },
        "onDone": {
          "target": "completed"
        },
        "onError": {
          "target": "failed"
        }
      }
    },
    "completed": {
      "type": "final"
    },
    "failed": {
      "type": "final"
    }
  }
}

How Execution Works

After the LLM returns the machine JSON, GenOpen runs this flow:

LLM machine JSON
↓
validate restricted XState subset
↓
create XState machine
↓
provide GenOpen actor: "tool"
↓
XState enters invoke state
↓
"tool" actor receives { tool, params }
↓
GenOpen resolves params, including $ref values
↓
registry.get(tool)
↓
validate params with tool.input Standard Schema
↓
handler({ input: parsedInput, signal })
↓
validate result.data with tool.output Standard Schema
↓
actor returns ToolResult
↓
XState follows onDone or onError
↓
GenOpen returns final status, final state, outputs

The LLM does not execute anything. It only emits the declarative machine. XState drives the machine, but GenOpen owns the bridge to registered tools.

Output and Reference Resolution

GenOpen must collect outputs by state id. Later states can reference prior outputs with $ref. This is not provided by XState automatically because it is a GenOpen workflow convention. Nested states use full state paths, for example createItems.createItem1.run.data.itemId. 

{ "$ref": "createMenu.data.menuId" }

Given this output:

{
  "createMenu": {
    "data": {
      "menuId": "menu:Dinner"
    }
  }
}

The ref resolves to:

"menu:Dinner"

Error and Retry Semantics

XState gives explicit error transitions through onError. If a tool throws, the actor rejects and XState can move to a failure state or retry path. 

{
  "createItem": {
    "invoke": {
      "src": "tool",
      "input": {
        "tool": "menu.item.create",
        "params": {
          "menuId": { "$ref": "createMenu.data.menuId" },
          "name": "Burger",
          "price": 12
        }
      },
      "onDone": { "target": "completed" },
      "onError": { "target": "failed" }
    }
  },
  "failed": {
    "type": "final"
  }
}

Retry can be represented either as explicit machine states or as safe metadata that GenOpen compiles into the machine. The safer initial approach is explicit states or a very small declarative retry policy. The LLM should not generate executable guards or actions.

Parallel Workflows

XState supports parallel states. This is useful for cases like creating ten menu items after the menu has been created. Once the menu id exists, each item creation can run independently. 

{
  "createItems": {
    "type": "parallel",
    "states": {
      "burger": {
        "initial": "create",
        "states": {
          "create": {
            "invoke": {
              "src": "tool",
              "input": {
                "tool": "menu.item.create",
                "params": {
                  "menuId": { "$ref": "createMenu.data.menuId" },
                  "name": "Burger",
                  "price": 12
                }
              },
              "onDone": "done",
              "onError": "failed"
            }
          },
          "done": { "type": "final" },
          "failed": { "type": "final" }
        }
      }
    },
    "onDone": "completed"
  }
}

What We Need to Test

We do not test XState itself. We test GenOpen’s integration with it. The tests should treat the LLM response as already available machine JSON.

Initial TDD slices:

  1. Single invoke machine executes one registered tool.
  2. Two-state machine passes output through $ref.
  3. Unknown tool fails predictably.
  4. Tool error follows onError to a failed final state.
  5. Parallel state invokes multiple tools after a dependency has completed.

Example test shape:

Deno.test("runWorkflow executes XState tool actor machine", async () => {
  const registry = createToolRegistry([createMenu, createMenuItem]);

  const workflow = {
    id: "menuWorkflow",
    initial: "createMenu",
    states: {
      createMenu: {
        invoke: {
          src: "tool",
          input: {
            tool: "menu.create",
            params: { name: "Dinner" }
          },
          onDone: { target: "createBurger" }
        }
      },
      createBurger: {
        invoke: {
          src: "tool",
          input: {
            tool: "menu.item.create",
            params: {
              menuId: { "$ref": "createMenu.data.menuId" },
              name: "Burger",
              price: 12
            }
          },
          onDone: { target: "completed" }
        }
      },
      completed: { type: "final" }
    }
  };

  const result = await runWorkflow({ workflow, registry });

  assertEquals(result.status, "done");
  assertEquals(result.outputs.createBurger.data.menuId, "menu:Dinner");
});

Safe Subset

The LLM should not be allowed to emit full arbitrary XState config. GenOpen should define and validate a restricted subset.

Allowed initially:

  • id
  • initial
  • states
  • type: "final"
  • type: "parallel"
  • invoke.src: "tool"
  • invoke.input.tool
  • invoke.input.params
  • invoke.onDone
  • invoke.onError

Disallowed initially:

  • Inline JavaScript functions
  • Arbitrary actions
  • Arbitrary guards
  • Arbitrary actor implementations
  • Dynamic code references

Open Questions

  • Should retry be explicit machine states or GenOpen retry metadata compiled into XState?
  • Should output collection happen through actor wrappers or XState actions?
  • How rich should developer workflow preview events become: input resolved, output produced, duration?
  • How much XState config should the LLM be allowed to emit?
  • When do we introduce snapshot persistence for resume support?

Summary

The tool SDK remains thin and object-based. The workflow runtime should move toward XState. GenOpen’s job is not to invent a full workflow engine; it is to validate a safe LLM-generated machine, provide a tool actor, resolve references, validate tool contracts, execute runtime handlers, collect outputs, emit lifecycle events, and return a stable result.
