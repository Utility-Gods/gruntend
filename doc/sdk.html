GenOpen SDK

Usage Guide

GenOpen lets an app define tool contracts, expose those contracts to an LLM, receive an XState-style workflow JSON response, and execute that workflow with runtime handlers owned by the app.

Core idea: defineTool() is contract-only. Runtime execution happens later through closure handlers passed to runWorkflow().

Install / import

import {
  createGenOpenClient,
  defineTool,
  type WorkflowMachineConfig,
} from "genopen";
import * as v from "valibot";

1. Define tool contracts

A tool contract describes what the LLM is allowed to call. It includes a stable name, description, Standard Schema-compatible input schema, and Standard Schema-compatible output schema. 

const createMenu = defineTool({
  name: "menu.create",
  description: "Create a menu and return the created menu id.",
  input: v.object({
    name: v.string(),
  }),
  output: v.object({
    menuId: v.string(),
    name: v.string(),
  }),
  execution: "sequential",
});

const createMenuItem = defineTool({
  name: "menu.item.create",
  description: "Create one menu item inside an existing menu.",
  input: v.object({
    menuId: v.string(),
    name: v.string(),
    price: v.number(),
  }),
  output: v.object({
    itemId: v.string(),
    menuId: v.string(),
    name: v.string(),
  }),
  execution: "parallel",
});

const tools = [createMenu, createMenuItem] as const;

2. Create a client

const genopen = createGenOpenClient({
  tools,
});

The client owns the tool registry. It does not own app runtime state like cookies, auth tokens, database handles, or component state.

3. Receive workflow JSON from the LLM

The LLM returns declarative XState-style JSON. It does not return executable JavaScript. 

const workflow: WorkflowMachineConfig = {
  id: "create-menu",
  initial: "createMenu",
  states: {
    createMenu: {
      invoke: {
        src: "tool",
        input: {
          tool: "menu.create",
          params: { name: "Dinner" },
        },
        onDone: "createBurger",
        onError: "failed",
      },
    },
    createBurger: {
      invoke: {
        src: "tool",
        input: {
          tool: "menu.item.create",
          params: {
            menuId: { $ref: "createMenu.data.menuId" },
            name: "Burger",
            price: 12,
          },
        },
        onDone: "completed",
        onError: "failed",
      },
    },
    completed: { type: "final" },
    failed: { type: "final" },
  },
};

4. Run the workflow with closure handlers

Handlers are supplied at runtime, so they can close over request state, auth tokens, cookies, Svelte/React state, database clients, or any app-owned variables. 

const authToken = getAuthToken();
const tenantId = getTenantId();

const result = await genopen.runWorkflow(workflow, {
  handlers: {
    "menu.create": async ({ input, ok, err, signal, attempt, maxAttempts }) => {
      if (!authToken) {
        return err({
          code: "MISSING_AUTH_TOKEN",
          message: "Missing app auth token.",
        });
      }

      const menu = await api.createMenu(input, { authToken, tenantId, signal });

      return ok({
        menuId: menu.menuId,
        name: menu.name,
      });
    },

    "menu.item.create": async ({ input, ok, err, signal }) => {
      const item = await api.createMenuItem(input, { authToken, tenantId, signal });

      if (!item.ok) {
        return err({
          code: "MENU_ITEM_CREATE_FAILED",
          message: item.error.message,
          retryable: true,
        });
      }

      return ok({
        itemId: item.value.itemId,
        menuId: input.menuId,
        name: input.name,
      });
    },
  },
});

The handler receives a runtime context. input is already parsed by the tool input schema. ok(data) marks the invocation successful. err(reason) marks it as a controlled failure without throwing. attempt and maxAttempts describe the current retry state.

Schema-driven handler types

Handler input and output types come from the tool schemas. If the handler output does not match the declared output schema, TypeScript can catch it before runtime. 

// createMenuItem.output requires itemId, menuId, and name.
"menu.item.create": async ({ input, ok }) => {
  return ok({
    itemId: "item_1",
    menuId: input.menuId,
    // TypeScript error: name is missing
  });
}

Runtime also validates values with Standard Schema. This catches invalid LLM params and invalid handler output even if TypeScript is bypassed.

Runtime validation flow

resolve workflow params, including $ref
↓
validate params with tool.input schema
↓
call runtime handler with parsed input, ok, err, signal, attempt, maxAttempts
↓
if handler returns err(...), fail/retry/onError without storing output
↓
if handler returns ok(data), validate data with tool.output schema
↓
store successful output by state path
↓
continue workflow

Workflow result

type ToolResult<Output> =
  | { ok: true; data: Output }
  | { ok: false; error: ToolExecutionError };

type WorkflowRunResult = {
  status: "done" | "failed";
  // Only successful tool results are stored here by normal execution.
  outputs: Record<string, ToolResult<unknown>>;
  errors: Record<string, WorkflowError>;
  finalState: string;
};

Lifecycle events

GenOpen currently emits six runtime events through one callback: onEvent.

Event When it fires Payload
workflow.started Before the XState actor starts running. { workflowId }
tool.started After params are resolved and input schema validation passes, immediately before the handler runs. { workflowId, state, tool, input }
tool.completed After the handler returns ok(data) and output schema validation passes. { workflowId, state, tool, output: { ok: true, data } }
tool.failed When input validation, handler execution, err(...), output validation, or tool lookup fails. { workflowId, state, tool, error }
workflow.completed When the workflow reaches a successful final state. { workflowId, finalState }
workflow.failed When the workflow reaches the failed final state. { workflowId, finalState, errors }

Using events for UI progress

const statuses: Record<string, "pending" | "running" | "done" | "error"> = {};

await genopen.runWorkflow(workflow, {
  handlers,
  onEvent(event) {
    if (event.type === "tool.started") {
      statuses[event.state] = "running";
    }

    if (event.type === "tool.completed") {
      statuses[event.state] = "done";
    }

    if (event.type === "tool.failed") {
      statuses[event.state] = "error";
    }
  },
});

Parallel workflows

XState parallel states run child regions concurrently. GenOpen records nested outputs by full state path. 

{
  "createItems": {
    "type": "parallel",
    "states": {
      "createItem1": {
        "initial": "run",
        "states": {
          "run": {
            "invoke": {
              "src": "tool",
              "input": {
                "tool": "menu.item.create",
                "params": {
                  "menuId": { "$ref": "createMenu.data.menuId" },
                  "name": "Burger",
                  "price": 12
                }
              },
              "onDone": "done",
              "onError": "failed"
            }
          },
          "done": { "type": "final" },
          "failed": { "type": "final" }
        }
      }
    },
    "onDone": "completed"
  }
}

Output key example:

outputs["createItems.createItem1.run"].data.itemId

Mental model

defineTool = contract
createGenOpenClient = registry/client
LLM workflow JSON = declarative state machine
handlers = runtime closures returning ok(data) or err(reason)
runWorkflow = validation + retry + XState execution + events
